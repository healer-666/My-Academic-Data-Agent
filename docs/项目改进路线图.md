# Academic-Data-Agent 项目改进路线图

这份路线图只关注当前版本之后最值得继续做的方向。

## 1. 当前状态判断

当前项目已经具备比较完整的工程闭环：

- 结构化表格数据输入
- 受控分析 workflow
- 工程化 RAG
- 证据归因
- reviewer 治理
- Project Memory
- Web 工作台、历史回看与历史问答

因此下一步重点不应该是继续堆入口，而应该是：

1. 提升可评估性
2. 提升可信度
3. 提升历史问答质量

## 2. 第一优先级：历史问答评测

既然项目已经新增历史问答主线，最值得优先补的是评测体系。

建议补齐：

- 固定历史运行样本集
- 固定追问问题集
- 单次追问与跨运行对比的对照测试
- 比较只读报告、读全量工件、读全量工件加记忆三种模式

为什么优先：

- 当前“分析 + 历史追问”已经形成新主线
- 没有评测就很难判断历史问答到底是检索问题、提示词问题还是工件组织问题

## 3. 第二优先级：历史知识切片进一步结构化

当前历史问答已经能从报告、轨迹、审稿和记忆里取上下文，但还可以更细：

- 报告按章节切片
- 审稿按轮次切片
- 轨迹按步骤类型切片
- 图表补更多说明字段
- 将关键统计量单独抽为结构化记录

这一层做好之后，历史问答会更像“分析档案检索”而不是“读大段文本再总结”。

## 4. 第三优先级：证据归因继续增强

当前已经有 evidence register 和 reviewer 校验，下一步可以继续增强：

- 更细粒度的段落级引用覆盖分析
- 对历史问答回答也做来源约束
- 对未通过审稿的运行增加更显式的风险提示

## 5. 第四优先级：Memory v2

当前 Project Memory 已经可用，建议下一步做得更稳，而不是更杂。

建议方向：

- 明确区分 `user_preference / review_constraint / analysis_pattern / retrieval_note`
- 支持查看某个 scope 的 memory 摘要
- 将 rejected run 的经验与 accepted run 分开存放

## 6. 第五优先级：Web 工作台继续产品化

当前工作台已经够用，下一步更适合做细化，而不是加更多入口。

建议方向：

- 历史问答来源跳转更清晰
- 历史运行筛选更丰富
- 运行结果页补更多工件摘要卡片
- 更直观地展示审稿与记忆写回状态

## 7. 当前不建议优先做的事

- 再把 PDF 拉回正式主线
- 通用聊天 Agent
- 大而全的多工具平台化
- 多租户平台架构
- 重型基础设施重构

## 8. 一句话结论

下一步最值得做的，不是再加一个花哨入口，而是把“历史问答、评测、证据约束和产品细节”继续做深。
