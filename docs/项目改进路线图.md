# Academic-Data-Agent 项目改进路线图

这份路线图只关注一件事：

> 在当前版本已经具备完整主链的前提下，下一步最值得继续做什么？

先说总判断：

当前项目已经具备比较完整的工程闭环，所以下一步重点不应该是继续加入口，而应该是把“评测、可信度、历史问答质量和用户体验”做深。

---

## 1. 当前状态判断

2026-04-24 更新：

- harness 已完成一轮 `seed_v4` 级别的阶段性验证，当前达到 `7/10 accepted`
- `two_group_small_sample`、`correlation_without_causality`、`time_series_trend_clean`、`reference_guideline_lookup`、`memory_constrained_repeat_task` 的扩展 smoke 均已通过
- 剩余主要问题集中在三个 `cleaning_contract_failure` 任务：`missing_values_by_group`、`outlier_sensitive_measurement`、`multi_group_with_variance_shift`
- 下一步不建议继续扩大 prompt 面积，而应逐个检查失败 run 并做定点修复

当前项目已经具备这些完整能力：

- 结构化表格输入
- 受控分析工作流
- 外部参考资料检索
- 证据登记
- 阶段执行审计
- reviewer 治理
- 成功经验 / 失败教训分层沉淀
- Web 工作台、历史回看与历史问答

因此下一步更值得补的是：

1. 可评测性
2. 可解释性
3. 历史问答质量
4. 用户体验

---

## 2. 第一优先级：历史问答评测

既然现在项目已经形成“分析 + 历史追问”的新主线，最应该优先补的是历史问答评测。

建议补齐：

- 固定历史运行样本集
- 固定追问问题集
- 单次追问与跨运行对比的对照测试
- 比较“只读报告 / 读运行档案 / 读运行档案加记忆”三种模式的表现

为什么这件事最值钱？

- 因为历史问答已经不是边角功能，而是当前正式能力的一部分
- 没有评测，就很难判断问题到底出在检索、切片、提示词还是来源组织上

---

## 3. 第二优先级：记忆质量治理

当前记忆已经分成成功经验和失败教训两路，这是正确方向；下一步最值得补的是“治理能力”，而不是继续堆量。

建议方向：

- 给成功经验和失败教训补更多可解释元数据
- 支持按 `scope` 查看某类任务当前沉淀了什么经验
- 支持弱化、删除或重建低质量记忆
- 统计哪些记忆经常被命中、哪些几乎没价值

为什么重要？

- 因为记忆库是会越用越大的
- 如果只写不管，后面一定会变乱

---

## 4. 第三优先级：历史问答切片结构化

当前历史问答已经能从报告、轨迹、审稿、图表摘要和记忆里取上下文，但还可以做得更细。

建议补强：

- 报告按章节切片
- 审稿按轮次切片
- 轨迹按步骤类型切片
- 图表补更多结构化说明字段
- 把关键统计量、方法名、结论类型抽成结构化片段

这样做之后，历史问答会更像“分析档案检索”，而不是“拿几大段文本再让模型总结”。

---

## 5. 第四优先级：证据与来源展示增强

当前已经有 evidence register 和 reviewer 校验，但前端对“来源”的展示还可以继续加强。

建议方向：

- 历史问答答案里更直观地列出来源类型
- 明确区分来源是运行档案、成功经验、失败教训还是外部参考资料
- 对非 `accepted` 的来源给出更醒目的状态提示
- 让用户更方便跳回关联报告、图表或审稿记录

这件事很适合继续强化项目的“可信度设计”。

---

## 6. 第五优先级：Web 工作台产品化细化

当前前端已经比之前更面向用户，但还可以继续往“分析工作台”打磨，而不是增加更多技术选项。

建议方向：

- 知识库状态页做得更清楚
- 历史记录筛选更丰富
- 结果页总结卡片更聚焦
- 历史问答来源跳转更顺
- 让阶段审计、审稿状态和记忆写回状态的呈现更直观

关键原则是：

> 少讲技术名词，多展示用户真正关心的事情：上传了什么、分析到了哪一步、结论是什么、靠不靠谱、之后还能不能追问。

---

## 7. 第六优先级：分析质量评测

除了历史问答，分析主链本身也值得补更系统的评测。

建议补：

- 固定表格数据样本集
- 固定问题模板
- 固定预期结论或关键统计指标
- 统计 reviewer 打回率、阶段审计失败率和最终通过率
- 比较开关成功经验、失败教训、RAG 前后的差异

这样可以把项目从“能跑”进一步拉向“可比较、可验证”。

---

## 8. 当前不建议优先做的事

- 把 PDF 再拉回正式主线
- 把项目包装成通用聊天 Agent
- 继续堆很多新工具入口
- 做很重的平台化基础设施重构

原因很简单：

- 这些方向会让主线重新变散
- 而当前最有价值的恰恰是“主线清楚”

---

## 9. 一句话结论

下一步最值得做的，不是再加一个花哨入口，而是把“历史问答评测、记忆治理、来源展示和工作台体验”继续做深，让这条“表格分析 + 历史追问”的主线更稳、更可信、更像一个完整产品。
