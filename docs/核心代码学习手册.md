# Academic-Data-Agent 核心代码学习手册

这份手册面向两个目标：

1. 帮你快速建立对当前项目真实状态的正确理解
2. 帮你在答辩、简历、面试或汇报里把项目讲清楚

如果只记一句话，可以记成：

> 这是一个以结构化表格分析为主线、把检索增强、阶段执行审计、审稿治理、成功经验 / 失败教训沉淀和历史追问串起来的分析型智能体系统。

---

## 1. 先把项目定位讲准

当前项目最准确的定位是：

> 一个基于 `hello-agents` 二次开发的分析型智能体工作流系统，正式主线聚焦结构化表格数据分析，并支持对历史分析结果继续追问。

它不是什么？

- 不是通用聊天机器人
- 不是通用 Agent 平台
- 不是以 PDF 解析为主角的文档理解系统
- 不是一个工具特别多的多代理协作平台

为什么这个定位重要？

因为这决定了你后面怎么讲项目的亮点：

- 要强调“工作流治理和可信度设计”
- 不要把重点讲成“我接了很多输入形态”

## 3. 先抓总链路

当前主链路可以先背成这一句：

```text
表格输入 -> 数据上下文 -> 成功经验 / 失败教训 -> 外部参考资料检索 -> ReAct 分析循环 -> 阶段执行审计 -> 审稿 -> 成功 / 失败记忆写回 -> 历史追问
```

如果你能把这条线讲清楚，项目就已经讲对了大半。

## 4. 核心模块分别在干什么

### `agent_runner.py`

职责：

- 承接 CLI / Web / Python API
- 组织整条运行时工作流
- 串联数据上下文、记忆、RAG、分析、审计、审稿和写回
- 返回统一的 `AnalysisRunResult`

理解重点：

- 它是 orchestration 层
- 大部分“项目到底怎么跑”的答案都在这里

### `data_context.py`

职责：

- 把原始表格压缩成模型可快速理解的数据摘要
- 提供 schema、shape、样例和数据警告

理解重点：

- 这里不是正式分析
- 它只是帮助模型看懂数据长什么样

### `knowledge_context.py`

职责：

- 把 user intent、成功经验、失败教训、RAG 结果和证据登记拼成注入上下文

理解重点：

- 这是项目的“上下文装配层”
- 很多你在问题里问到的“这些信息最后怎么进 prompt”都要看这里

### `execution_audit.py`

职责：

- 检查是否明确生成 `cleaned_data.csv`
- 检查后续正式分析步骤是否明确重读这个文件

理解重点：

- 它是从“软约束”走向“硬校验”的关键模块
- 是这个项目可信度设计里非常值得讲的一点

### `rag/service.py`

职责：

- 处理外部参考资料的索引、切块、检索、重排与证据登记

理解重点：

- RAG 是“外部依据层”
- 它不是项目内部经验库

### `memory/service.py`

职责：

- 管理成功经验检索与失败教训检索
- 提供独立的格式化输出，分别注入 prompt

理解重点：

- 当前的 memory 已经不是一个桶
- 它被拆成“正向经验”和“负向约束”两路

### `memory/extractor.py`

职责：

- 从成功运行中提炼成功经验
- 从完整失败运行中提炼失败教训

理解重点：

- 写回不是把原始报告整份丢进去
- 而是抽取出更适合长期复用的记录

### `history_qa.py`

职责：

- 从历史工件、成功经验和失败教训中构建问答切片
- 支持单次追问和跨运行对比

理解重点：

- 它和分析智能体是分开的
- 只读历史，不重新跑新分析

### `review_service.py`

职责：

- 构建 reviewer 任务
- 注入报告、轨迹、证据和记忆上下文
- 产出结构化审稿结果

理解重点：

- reviewer 是治理层，不是第二个自由分析 Agent

### `artifact_service.py`

职责：

- 落盘报告、图表、trace 和审稿记录
- 汇总运行元数据

理解重点：

- 这个项目很强调“可回放”和“可复盘”

## 5. 你最应该真正理解的几个问题

### 5.1 为什么 `data_context` 不是全量数据，结果却还能基于全量数据？

因为它是两层设计：

- `data_context` 只负责让模型快速理解数据结构
- Python 工具才负责真正读取全量数据做计算

现在又加上了阶段执行审计，所以不是单纯靠 prompt 约束，而是会真正检查：

- 有没有生成 `cleaned_data.csv`
- 有没有重读 `cleaned_data.csv`

### 5.2 analyst loop 是不是 plan-and-execute？

不是。

当前主循环更接近 `ReAct`：

- 每次只决定下一步做什么
- 调工具
- 看结果
- 再继续推进

外层虽然有审稿返修循环，但不等于项目内部已经有独立的 planner 和 executor 两个角色。

### 5.3 联网搜索结果是怎么进来的？

如果模型在 ReAct 过程中主动调用联网搜索工具，结果会作为工具观察结果被压缩后回注到下一轮消息里。

所以联网结果不是在最开始一次性塞进 prompt，而是**作为工具调用结果动态注入**。

### 5.4 RAG、成功经验、失败教训、运行档案到底是什么关系？

这是现在项目里最容易混的地方，建议你直接记成四层：

- **运行档案**：每次运行都保留，是真实原始工件
- **外部参考资料库**：给 RAG 用，存外部资料
- **成功经验库**：只存通过审稿后值得复用的经验
- **失败教训库**：只存完整失败后形成的负向约束

历史问答则主要读运行档案，再补读成功经验和失败教训。

### 5.5 为什么失败经验也要存，但不能和成功经验混在一起？

因为失败经验确实值得记住，但它的用途不是“下次照着做”，而是“下次别再这么做”。

所以当前设计是：

- 成功经验：正向提示
- 失败教训：负向约束

这个区分很重要，也很适合在汇报里展示你对系统设计边界的理解。

## 6. 当前最值得你拿出去讲的技术点

如果你要对老师、面试官或同学解释项目，最值得讲的是这几件事：

1. 为什么这是一个“受控分析工作流”，不是普通聊天机器人
2. 为什么把外部参考资料、成功经验、失败教训和运行档案分层
3. 为什么要做阶段执行审计，确保正式分析真基于全量清洗数据
4. 为什么要做 evidence register 和 reviewer 证据检查
5. 为什么历史问答是“读取历史工件继续追问”，而不是重新执行分析

这些问题都比“我用了哪些库”更能体现项目深度。

## 7. 当前最容易讲错的地方

### 不要说成“通用 Agent 平台”

因为当前工具不算多，定位也不是平台化。

### 不要把 PDF 当成主卖点

现在正式主线已经收束到结构化表格分析，PDF 只是旧兼容痕迹，不是当前核心竞争力。

### 不要把 RAG 和记忆混成一个东西

RAG 负责外部依据，记忆负责内部经验，两者不是一回事。

### 不要说“模型看几行摘要就生成了完整分析”

这会显得项目很虚。正确说法是：

> 模型先用摘要理解数据，再通过 Python 读取全量文件完成正式分析，并接受阶段执行审计。
