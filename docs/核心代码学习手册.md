1. 快速建立对当前代码主线的正确理解

## 1. 先明确项目定位

当前项目最准确的定位是：

> 一个基于 `hello-agents` 二次开发的学术数据分析 Agent 工作流系统，正式主线聚焦结构化表格分析，并支持对历史分析结果继续追问。

它不是：

- 通用聊天机器人
- 通用 Agent 平台
- 以 PDF 解析为主角的文档理解系统

## 3. 主链路一句话概括

```text
上传表格 -> 数据上下文 -> Project Memory 回忆 -> RAG 检索 -> analyst 分析循环 -> 报告 / trace -> reviewer -> accepted run 写回 memory -> 历史问答
```

## 4. 关键模块各自负责什么

### `agent_runner.py`

职责：

- 作为稳定门面承接 CLI / Web / Python API
- 组装整条运行时工作流
- 串联 data context、RAG、memory、report、review、artifact
- 返回统一的 `AnalysisRunResult`

理解重点：

- 它是 orchestration 层
- 当前最重要的业务主线都从这里进入

### `data_context.py`

职责：

- 将输入表格压缩成模型可消费的数据摘要
- 提供 schema、shape、head sample、warning 等信息

理解重点：

- 当前项目不是把整份原始表直接塞给模型
- 而是先做一层可控的数据摘要

### `knowledge_context.py`

职责：

- 统一拼装 user intent、memory、reference context、retrieved knowledge、evidence register

理解重点：

- 它是 prompt 注入层
- RAG 和 memory 都通过它进入分析阶段

### `rag/`

职责：

- 文档读取
- structured chunking
- 向量检索
- 关键词检索
- 混合重排
- evidence register

理解重点：

- 当前 RAG 的定位是“表格分析的知识补充层”和“历史问答的检索底座”

### `memory/`

职责：

- 生成 `memory_scope_key`
- 回忆项目级偏好、约束和经验
- 将 accepted run 的经验写回长期记忆

理解重点：

- 当前 memory 是项目级经验层，不是通用会话记忆系统

### `history_qa.py`

职责：

- 从历史运行工件构建问答知识记录
- 做历史切片检索
- 回答单次追问和跨运行对比问题

理解重点：

- 它和分析智能体分离
- 只读取历史工件，不重新执行新分析代码

### `review_service.py`

职责：

- 构建 reviewer 任务
- 注入 report、trace、evidence、memory 等上下文
- 产出结构化 review log

理解重点：

- reviewer 是治理层，不是第二个自由分析 Agent

### `artifact_service.py`

职责：

- 落地报告、trace、review logs、工件包
- 汇总关键运行元数据

理解重点：

- 这个项目强调“可回放、可复盘、可下载”

## 5. 现在最值得你会讲的技术点

- 为什么不用纯聊天，而是做受控 workflow
- 为什么把 RAG、reviewer、memory、history QA 分层
- 为什么要做 evidence register 和审稿校验
- 为什么只有 accepted run 才写入 Project Memory
- 为什么历史问答只读历史工件，不直接重新分析

## 6. 一句话结论

如果要把当前项目讲清楚，最稳的说法是：

> 这是一个以结构化表格分析为主线，结合 RAG、证据归因、审稿治理、项目记忆和历史问答的分析型 Agent 系统。
