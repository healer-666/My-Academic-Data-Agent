# Academic-Data-Agent 核心代码学习手册

这份手册面向两个目标：

1. 快速建立对当前代码主线的正确理解。
2. 帮助你在简历、答辩或面试里把项目讲清楚。

## 1. 先明确项目定位

当前项目最准确的定位是：

> 一个基于 `hello-agents` 二次开发的学术数据分析 Agent 工作流系统。

它不是通用聊天机器人，也不是通用 MCP 平台。它的核心价值在于把下面这些能力串成一条完整闭环：

- 表格 / PDF 输入接入
- 受控分析工作流
- RAG 检索增强
- 证据归因与引用约束
- Reviewer 审稿治理
- Project Memory 项目级记忆
- Web 工作台与历史回放

## 3. 主链路一句话概括

```text
输入文件 -> 文档解析 / 数据上下文 -> Project Memory 回忆 -> RAG 检索 -> analyst 分析循环 -> 报告 / trace -> reviewer -> accepted run 写回 memory
```

## 4. 关键模块各自负责什么

### `agent_runner.py`

职责：

- 作为稳定门面承接 CLI / Web / Python API
- 组装整条运行时工作流
- 串联 data context、RAG、memory、report、review、artifact
- 返回统一的 `AnalysisRunResult`

理解重点：

- 它是 orchestration 层，不是单纯脚本。
- 当前复杂逻辑已经逐步拆入 service，但主入口仍然在这里。

### `document_ingestion.py`

职责：

- 处理 PDF 输入
- 抽取背景文本
- 识别 candidate tables
- 选择主表
- 生成 `parsed_document.json`

理解重点：

- 当前 PDF 路线是“一张主表做正式定量分析，其他候选表做上下文增强”。
- 它不是通用 PDF 平台，也不处理复杂 OCR 流程。

### `data_context.py`

职责：

- 将输入文件压缩成模型可消费的数据摘要
- 提供 shape、schema、head sample、warning、主表信息等

理解重点：

- 这一步决定模型看到的数据规模和结构感。
- 当前项目不是把整份原始表直接塞给模型。

### `knowledge_context.py`

职责：

- 统一拼装用户意图、memory、reference context、retrieved knowledge、evidence register

理解重点：

- 它是 prompt 注入层。
- 当前 RAG 和 memory 都是通过这里进入分析阶段。

### `rag/`

职责：

- 文档读取、structured chunking、向量索引、关键词索引、hybrid retrieval、rerank、evidence register

理解重点：

- 当前已经不是“只有向量库”的初级 RAG。
- 已支持 `query rewrite + hybrid retrieval + structured chunking + evidence attribution`。

### `memory/`

职责：

- 生成 `memory_scope_key`
- 从 accepted run 提取项目级经验
- 在分析前 / 审稿前回忆项目级偏好与约束

理解重点：

- 当前 memory 是“项目级精炼经验”，不是完整会话记忆网络。
- 只写回 accepted run，避免把噪音灌进长期记忆。

### `review_service.py`

职责：

- 构建 reviewer 任务
- 注入 report、trace、evidence、memory 等上下文
- 产出结构化 review log

理解重点：

- reviewer 是治理层，不是第二个自由分析 agent。
- 当前还带有证据一致性硬校验。

### `artifact_service.py`

职责：

- 落地报告、trace、review logs、工件包
- 汇总关键运行元数据

理解重点：

- 这个项目强调“可回放、可复盘、可下载”，不是只返回一段文本。

## 5. 现在最值得你会讲的技术点

- 为什么不用纯聊天，而是做显式 workflow
- 为什么把 PDF ingestion、RAG、reviewer、memory 分层
- 为什么要做 evidence register 和行内短引用
- 为什么 accepted run 才写入 project memory
- 为什么 Web 要支持历史记录和 trace 回看

## 6. 一句话结论

如果要把当前项目讲清楚，最稳的说法是：

> 这是一个面向学术数据与论文 PDF 的分析型 Agent 系统，核心不在“聊天”，而在“受控工作流 + RAG + 证据归因 + 审稿治理 + 项目记忆”的工程闭环。
