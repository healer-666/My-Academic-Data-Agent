# Academic-Data-Agent 令牌、上下文与审稿说明

这份文档专门回答四个容易被误解的问题：

1. 当前项目到底有没有做系统级的令牌管理
2. 为什么要压缩工具观察结果
3. 当前 prompt 里到底塞了哪些层的上下文
4. reviewer 到底依据什么来判断报告能不能过

---

## 1. 当前做到了什么程度的令牌管理

先说结论：

> 当前项目没有做成一个完整的“全局令牌调度平台”，但已经做了足够支撑稳定分析主链的上下文体积控制。

也就是说，它不是那种带精细预算分配、全局裁剪策略、动态窗口管理的重型系统；但它已经不是“把所有东西不加控制地往 prompt 里堆”。

当前已经做的，主要有三类：

### 1.1 tokenizer 兼容

在 [`config.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/config.py) 中，项目对 `hello_agents` 的计数器做了兼容处理，避免 OpenAI-compatible 接口下 tokenizer 不匹配带来的问题。

它解决的是：

- 模型接入层的兼容问题

它没有解决的是：

- 多阶段任务之间如何做更精细的全局预算分配

### 1.2 数据上下文压缩

通过 [`data_context.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/data_context.py)，原始表格不会直接整张塞进 prompt，而是压缩成：

- 字段信息
- 类型
- 规模
- 样例行
- 数据警告

### 1.3 工具观察结果压缩

分析主循环中，工具返回结果不会原样全部回注，而是会做摘要回注；完整原文则落到 trace 或日志里。

这点对稳定性非常重要。

## 2. 为什么要压缩 observation

当前 analyst loop 是多步的，结构大致是：

```text
模型决策 -> 调工具 -> 得到 observation -> 将 observation 回注 -> 再决策
```

如果 observation 不压缩，会有两个直接问题。

### 2.1 prompt 会迅速膨胀

Python 工具可能输出：

- 很长的 stdout
- traceback
- 统计表
- 图表路径
- 中间分析说明

如果全部原样回注，几步之后上下文就会非常重。

### 2.2 成本和稳定性都会变差

上下文越长：

- 速度越慢
- 成本越高
- 模型越容易丢焦点
- 后续决策越容易跑偏

### 当前策略是什么？

当前采用“双轨策略”：

- **给模型看的**：压缩后的 observation 摘要
- **给人复盘的**：更完整的 trace / logs

也就是说，系统并不是丢掉细节，而是把“模型需要的”和“人类排查需要的”分开存放。

## 3. 当前上下文到底分了哪些层

这是最值得真正搞清楚的部分。

现在项目不是只有一个模糊的大 prompt，而是把不同来源的信息分层注入。

### 3.1 `data_context`

来自输入表格，主要负责回答：

- 这是什么数据
- 有哪些字段
- 规模大概多大
- 一眼看上去有什么风险

它不是正式分析结果，也不是证据来源。

### 3.2 `<Success_Memory_Context>`

来自成功经验库，主要包括：

- 过去稳定有效的分析做法
- 用户偏好
- 已验证的 reviewer 约束

它回答的是：

- 这类任务以前怎样做更稳

### 3.3 `<Failure_Memory_Context>`

来自失败教训库，主要包括：

- 不要再犯的错误
- 容易被打回的模式
- 开始前必须额外检查的事项

它回答的是：

- 这类任务这次不要怎么做

很关键的一点是：

> 失败教训只作为负向约束，不作为事实依据或外部知识来源。

### 3.4 `Retrieved_Knowledge_Context`

来自 RAG 检索，内容是：

- 已收录参考资料中最相关的片段

它回答的是：

- 这次分析有哪些可引用的背景材料和外部依据

### 3.5 `Retrieved_Evidence_Register`

这是对证据片段的结构化整理，通常包含：

- evidence id
- 引用标签
- 来源定位信息

它回答的是：

- 报告里的知识性解释应该引用哪条证据

### 3.6 `History QA Context`

历史问答阶段会额外读取：

- 历史报告摘要
- 审稿摘要
- 轨迹摘要
- 图表摘要
- 成功经验片段
- 失败教训片段

它的作用不是做新分析，而是围绕历史运行继续解释和比较。

## 4. `data_context` 只是摘要，那怎么保证最后真的基于全量数据？

这是你之前问过的核心问题之一。

现在项目的设计是：

1. `data_context` 只负责帮助模型理解数据结构
2. 正式分析必须走 Python 工具读取本地文件
3. 阶段执行审计会检查是否明确生成并重读 `cleaned_data.csv`

也就是说，现在不是只靠 prompt 说一句“请基于全量数据分析”，而是会真正检查：

- 有没有保存清洗后的文件
- 后续步骤有没有明确重新读取它

如果不能证明这件事，就不能通过。

## 5. reviewer 到底依赖什么

reviewer 不是只读一遍报告就拍板，它的判断是建立在多种输入共同作用上的。

当前 reviewer 会综合这些内容：

- 最终报告
- 关键轨迹摘要
- 关键工件校验结果
- 阶段执行审计结果
- evidence register
- 引用覆盖情况
- 成功经验和失败教训

### reviewer 重点看什么？

- 结论是不是过度推断
- 图表是否支撑结论
- 知识性解释有没有来源
- 引用和证据是否匹配
- 是否违反了历史上已经被验证过的约束
- 是否通过了阶段执行审计

所以 reviewer 的角色更像“审稿人 + 质检”，而不是另一个自由发挥的分析助手。

---

## 6. 联网搜索结果怎么进上下文

这也是很多人容易误解的地方。

如果 analyst loop 中启用了联网工具，模型在某一步主动调用它，那么：

1. 搜索工具返回结果
2. controller 把结果压缩成 observation summary
3. 这个 summary 被回注到下一轮消息中

所以联网结果不是在分析前一次性塞进去，而是作为 ReAct 工具调用结果**动态进入上下文**。

---

## 7. 一句话结论

当前项目没有重型的全局令牌平台，但已经形成了一套足够清晰的上下文治理方式：

- 表格摘要负责理解数据
- 成功经验负责正向提示
- 失败教训负责负向约束
- RAG 负责外部依据
- 证据登记负责引用落点
- 审稿和阶段执行审计负责最后把关

这几层叠在一起，才构成了当前比较稳的分析主链。
