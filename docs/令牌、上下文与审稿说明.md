# Academic-Data-Agent 令牌、上下文与审稿说明

这份文档专门回答三个容易被误解的问题：

1. 项目现在有没有做令牌管理
2. 为什么要压缩 observation
3. reviewer 的判断到底依赖什么

## 1. 令牌：当前做到什么程度

当前项目没有完整的 token budgeting 平台，也没有复杂的全局上下文配额调度系统。

已经做的主要是两类：

### 1.1 tokenizer 兼容 patch

在 [`config.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/config.py) 中，项目对 `hello_agents` 的 token counter 做了 OpenAI-compatible fallback patch。

它解决的是：

- OpenAI-compatible 模型接入时的 tokenizer 兼容问题

它没有解决的是：

- 全局 token 成本调度
- 多阶段上下文预算优化

### 1.2 上下文体积控制

项目会主动控制这些内容的 prompt 体积：

- `data_context`
- `knowledge_context`
- retrieved chunks
- evidence register
- memory context
- tool observation summaries
- history QA 检索切片

更准确的说法是：

> 当前项目已经做了 tokenizer 兼容和上下文体积控制，但还没有形成完整的 token 管理平台。

## 2. 为什么要压缩 observation

主链是多步的：

```text
模型决策 -> 调工具 -> 得到 observation -> observation 回注 -> 再决策
```

如果 observation 不压缩，会有两个直接问题：

### 2.1 上下文迅速膨胀

Python 工具可能输出：

- 大量 stdout
- traceback
- 表格统计
- 图表路径

如果全量回注，几轮之后 prompt 会非常重。

### 2.2 成本与稳定性下降

上下文越长：

- 模型越慢
- token 越贵
- 任务越容易跑偏

### 当前策略

当前项目采用“双轨策略”：

- 原始 observation 保留在 trace / logs
- 压缩摘要回注给下一轮模型

也就是说：

- 给模型看的，是足够用的摘要
- 给人排查问题看的，是更完整的执行痕迹

## 3. 当前上下文已经分了哪些层

当前不是只有一个大 prompt，而是多层上下文：

### `data_context`

来自输入表格：

- schema
- shape
- sample
- warning

### `Project_Memory_Context`

来自 accepted run 的项目级经验：

- 用户偏好
- 审稿约束
- 历史分析模式

### `Retrieved_Knowledge_Context`

来自知识库检索：

- 最终 reranked chunks

### `Retrieved_Evidence_Register`

来自最终检索证据：

- evidence id
- citation label
- source locator

### `History QA Context`

来自历史运行工件：

- report summary
- review summary
- trace summary
- figure summaries
- memory snippets

## 4. reviewer 到底依赖什么

reviewer 不是随便读一遍报告就给结论，而是综合这些输入：

- 最终报告
- 关键 trace
- 产物校验结果
- evidence register
- 实际引用情况
- project memory

因此 reviewer 判断的重点是：

- 报告是否稳妥
- 是否满足质量档位要求
- 是否存在缺引用、伪造引用或证据错配
- 是否违背项目级约束

## 5. 一句话结论

当前项目没有完整的 token 平台，但已经有：

- 上下文体积控制
- observation 摘要回注
- 分层上下文注入
- reviewer 证据校验

这些已经足以支撑一条比较稳定的分析型 Agent 主链。
