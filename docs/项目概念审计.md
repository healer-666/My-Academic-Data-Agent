# Academic-Data-Agent 项目概念审计

这份文档只回答一个问题：

> 当前项目到底已经正式实现了什么，哪些还只是未来方向？

写这份文档的目的，是避免把项目讲得太虚，也避免把已经做出来的东西讲轻了。

---

## 1. 判断标准

只有同时满足下面两个条件，才算“当前版本正式实现”：

1. 已经进入当前正式主链
2. 会对运行结果、轨迹、审稿或历史追问产生实质影响

因此：

- 目录存在，不等于能力已经成型
- 曾经试验过，不等于现在还是正式主线
- 仓库里留了兼容代码，不等于它仍然是项目卖点

---

## 2. 当前可以稳定宣称已经实现的能力

### 2.1 结构化表格分析主线

可以说：

- 当前正式输入主线是结构化表格数据分析
- 已经形成从上传到报告、审稿、记忆写回和历史追问的一条完整链路

不要说：

- 通用多模态输入平台

### 2.2 受控分析工作流

可以说：

- 有明确的主入口和受控运行状态
- analyst loop 采用 ReAct 风格多步分析
- 工具调用、观察结果回注、工件落盘和审稿返修都进入正式流程

不要说：

- 通用自主 Agent 平台

### 2.3 全量数据使用强校验

可以说：

- 正式分析前只给模型数据摘要
- 正式分析由 Python 读取全量文件完成
- 已加入阶段执行审计，检查 `cleaned_data.csv` 的生成和重读

不要说：

- 系统已经做了形式化验证级别的程序证明

因为当前仍是规则审计，不是静态分析器或沙箱级系统证明。

### 2.4 工程化 RAG

可以说：

- 有本地参考资料知识库
- 已实现查询改写、混合检索、结构化切块、重排和证据登记
- 可为分析报告和历史问答提供外部依据

不要说：

- 最前沿研究级通用 RAG 平台

### 2.5 证据归因

可以说：

- 已有 evidence register
- 报告中的知识性解释要求带短引用
- reviewer 会检查缺失引用、伪造引用和证据错配

不要说：

- 完整学术引文系统

因为当前引用更偏工程化证据约束，不是完整论文引文管理系统。

### 2.6 记忆分层

可以说：

- 已经把长期沉淀拆成成功经验、失败教训和外部参考资料三条线
- 成功经验只从可信运行中提炼
- 失败教训以负向约束形式参与后续分析

不要说：

- 通用长期会话记忆网络

### 2.7 历史问答

可以说：

- 已有独立历史问答页面
- 已支持围绕历史运行继续追问
- 已支持跨运行对比总结
- 会对非 `accepted` 的来源显式标注状态

不要说：

- 通用聊天问答 Agent

### 2.8 Web 工作台

可以说：

- 已有面向用户的分析工作台
- 支持上传、运行、结果查看、历史浏览和继续追问

不要说：

- 企业级复杂前端平台

当前前端更像聚焦分析任务的工作台，而不是大而全产品后台。

---

## 3. 当前已经弱化或退出正式主线的方向

### 3.1 PDF 主线

当前最准确的说法是：

- 仓库中仍保留部分兼容代码
- 但 PDF 已不再是当前版本的正式主线和主要卖点

因此不要再把项目介绍成“论文 PDF 理解系统”。

### 3.2 通用多工具平台化

当前工具层并不宽，重点在工作流治理，而不是工具生态规模。

因此不要再把它包装成：

- 通用多工具编排平台
- 通用多代理协作系统

### 3.3 通用聊天型 Agent

当前历史问答虽然是对话形态，但它的本质是：

- 基于历史工件的检索与解释

不是无边界自由聊天系统。

---

## 4. 当前仍然更适合说成“下一步方向”的内容

- 更细粒度的自动评测体系
- 更成熟的记忆管理界面
- 更丰富的历史问答来源跳转和引用展示
- 更细的证据覆盖率分析
- 更系统的前端产品化打磨

这些都值得做，但不该说成已经非常成熟。

---

## 5. 当前最稳的项目描述

当前最稳妥、也最接近实际代码状态的项目描述是：

> 这是一个面向科研和学术场景的结构化表格分析智能体系统，以受控分析工作流为主线，结合外部参考资料检索、成功经验与失败教训分层回忆、阶段执行审计、审稿治理、历史回放和历史问答，形成了一条可追溯、可解释、可复盘的分析流程。

---

## 6. 一句话结论

这个项目已经足够被描述成“有工程深度的分析型智能体系统”，但不应该再包装成“通用智能体平台”或“以 PDF 理解为核心的系统”。
