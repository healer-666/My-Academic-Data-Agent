# Academic-Data-Agent 项目概念审计

这份文档只回答一个问题：

> 当前项目到底已经实现了什么，哪些还只是未来方向。

## 1. 判断标准

只有满足下面两个条件，才能算“当前版本正式实现”：

1. 已经进入当前主链路。
2. 会对最终结果、trace 或审稿产生正式影响。

因此：

- 目录存在，不等于能力已实现。
- 试验过，不等于现在主链在用。
- 可以接入，不等于已经正式接入。

## 2. 当前可以稳定宣称的能力

### Agent 工作流

可以说：

- 项目基于 `run_analysis(...)` 驱动一条受控分析工作流
- 有显式状态和可回放的运行轨迹

不要说：

- 通用自治 Agent 平台

### PDF ingestion

可以说：

- 支持文本型 PDF 的背景提取、候选表识别、主表选择与结构化分析接入

不要说：

- 通用论文理解平台
- OCR 平台

### RAG

可以说：

- 已有本地知识库
- 已实现 `query rewrite + hybrid retrieval + structured chunking + rerank`
- 已支持 PDF 表格候选增强

不要说：

- 研究级最前沿 RAG 系统
- 完整多跳动态检索平台

### 证据归因

可以说：

- 已有 evidence register
- 报告中的知识性解释要求带行内短引用
- reviewer 会检查引用缺失、伪造引用和证据不匹配

不要说：

- 完整学术引文系统

### Project Memory

可以说：

- 已有项目级记忆
- accepted run 会被精炼写回
- 分析前和审稿前都可回忆

不要说：

- 完整长期记忆网络
- 通用会话记忆系统

### Reviewer 治理

可以说：

- 有文本 reviewer
- 有可选 visual reviewer
- 有 evidence consistency 检查

不要说：

- 多 agent 自由协作平台

## 3. 当前还没有正式进入主链的方向

- MCP 工具总线
- 通用知识问答 Agent 模式
- 复杂 cross-encoder / LLM reranker
- 成熟的 RAG 离线评测平台
- 完整长期多层记忆体系
- OCR 型 PDF 理解

## 4. 最稳的项目描述

当前最稳妥的项目描述是：

> 一个基于 `hello-agents` 二次开发的学术数据分析 Agent 系统，支持 PDF ingestion、工程化 RAG、证据归因、Reviewer 治理、Project Memory 和可回放 Web 工作台。

## 5. 一句话结论

这个项目已经足够被描述成“有技术深度的分析型 Agent 系统”，但不应该被包装成“通用智能体平台”。
