# Academic-Data-Agent 项目主链路拆解

这份文档只回答一个问题：

> 当前版本真正的正式主链路，到底是怎么跑的？

如果你只想先抓住全局，可以先记住这一句：

```text
上传表格 -> data_context -> 成功经验 / 失败教训检索 -> RAG 检索 -> analyst loop -> 阶段执行审计 -> report / trace -> review -> 记忆写回 -> 历史问答
```

这条链路里最容易被误解的点有三个：

1. `data_context` 不是“拿摘要直接分析”，它只是分析前的导航图。
2. analyst loop 主要是 `ReAct`，不是先写完整计划再执行的 plan-and-execute。
3. 长期沉淀不是一个库，而是“运行档案 / 外部参考资料 / 成功经验 / 失败教训”四层分开。

---

## 1. 起点：`run_analysis(...)`

真正的主入口是 [`run_analysis(...)`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/agent_runner.py)。

它负责把下面这些阶段串起来：

- 准备运行目录和配置
- 验证输入数据
- 构建 `data_context`
- 检索成功经验和失败教训
- 检索外部参考资料
- 运行 analyst loop
- 保存报告、图表和轨迹
- 执行阶段审计与 reviewer 审稿
- 写回成功经验或失败教训
- 返回统一的 `AnalysisRunResult`

通俗说，`agent_runner.py` 就是整个项目的“总调度台”。

---

## 2. 第一步：输入标准化

当前正式主线只支持：

- `csv`
- `xls`
- `xlsx`

也就是说，现在项目的主角是**结构化表格数据**。仓库里虽然还保留一些旧的文档解析兼容模块，但 PDF 已不再是当前版本对外强调的正式输入主线。

为什么这么做？

- 因为当前项目真正强的地方不是文档解析，而是后面的分析闭环
- 结构化表格输入更稳定，也更适合把工作流打磨深

---

## 3. 第二步：数据上下文构建

[`data_context.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/data_context.py) 会把原始表格压缩成一个适合模型快速理解的数据摘要，通常包括：

- 文件路径
- 列名
- 数据类型
- 行列规模
- 前几行样例
- 可能的数据警告

### 这一步到底有什么用？

它的作用不是“让模型直接拿这几行样本做正式分析”，而是：

- 让模型先知道这份数据大概是什么结构
- 帮它决定第一步 Python 代码该怎么写
- 控制上下文体积，避免把整张表塞进 prompt

### 你之前问过的关键问题：最后的报告和图表不是应该用全量数据吗？

对，正式分析必须基于全量数据。

这里的设计是两层：

1. `data_context` 负责“看懂数据”
2. Python 工具负责“真正读取全量文件做计算”

也就是说，模型先通过摘要决定怎么分析，但真正的统计量、检验结果和图表，必须由 Python 在本地重新读取数据后算出来。

---

## 4. 第三步：成功经验与失败教训检索

这是当前版本非常重要的一次升级。

现在主分析前不再只有一个笼统的 `Project Memory`，而是拆成了两路：

### 4.1 成功经验检索

来自 [`memory/service.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/memory/service.py) 中的成功经验服务，主要回忆：

- 这类任务以前什么做法比较稳
- 用户偏好怎样的输出风格
- reviewer 以前反复强调过哪些约束

它的角色是：**正向经验提示**。

### 4.2 失败教训检索

来自同一子系统中的失败教训服务，主要回忆：

- 哪些写法或步骤容易被 reviewer 打回
- 哪些分析风险需要提前规避
- 下次开始前必须额外检查什么

它的角色是：**负向约束和禁忌清单**。

### 为什么要分开？

因为失败经验很重要，但不能和成功经验混成一锅。

如果混用，模型就很容易把“曾经犯过的错误”也当成可复用经验。现在拆开后：

- 成功经验负责告诉模型“怎么做更稳”
- 失败教训负责提醒模型“不要再这样做”

---

## 5. 第四步：RAG 检索

当前 RAG 的职责已经很清楚了：**补外部依据，不补项目内部经验**。

它的链路大致是：

```text
查询改写 -> 向量检索 -> 关键词检索 -> 合并 -> 重排 -> 最终片段 -> 证据登记
```

RAG 现在主要用于三件事：

1. 给本次分析补充背景知识和参考资料
2. 给报告里的知识性解释提供证据片段
3. 给历史问答提供可检索的参考资料底座

### 你之前问过：如果开了联网搜索，搜索结果是怎么注入的？

这里要分开两种“外部信息”：

- **RAG 检索**：查的是本地知识库和已收录参考资料
- **联网搜索工具**：如果模型主动调用联网工具，搜索结果会作为工具观察结果回注到下一轮 analyst loop

也就是说，联网不是在主流程开始时一次性塞进 prompt，而是作为 ReAct 工具调用的一部分动态注入。

---

## 6. 第五步：Analyst Loop

这里是整个项目最核心的“行动层”。

analyst loop 的职责是：

- 读取当前上下文
- 决定下一步做什么
- 调用工具
- 读取工具返回结果
- 再决定下一步
- 直到完成报告

### 它到底是 ReAct 还是 plan-and-execute？

当前主循环本质上是 **ReAct**。

也就是说，它更像：

```text
思考 -> 调工具 -> 看结果 -> 再思考 -> 再调工具 -> 最后完成
```

而不是：

```text
先生成一整份完整计划 -> 再严格按计划机械执行
```

当然，系统 prompt 里会有阶段性要求，外面还有审稿返修循环，但这并不等于它是独立 planner + executor 的 plan-and-execute 架构。

### 当前最重要的工具是什么？

当前真正的核心执行工具其实很少，最重要的是：

- `PythonInterpreterTool`
- 可选的联网搜索工具

所以这个项目的强项不是“工具数量很多”，而是**分析流程治理得比较严**。

---

## 7. 第六步：阶段执行审计

这是最近一版非常关键的补强。

你之前问过一个很尖的问题：

> 虽然设计上要求正式分析基于全量数据，但怎么证明模型真的做到了，而不是偷懒只看摘要？

现在的答案是：**加了硬审计**。

[`execution_audit.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/execution_audit.py) 会检查两件事：

1. 有没有明确生成规范路径下的 `cleaned_data.csv`
2. 后续正式分析步骤里，有没有明确重新读取这个 `cleaned_data.csv`

如果系统只能“猜测”它做对了，而不能明确证明，就会按失败处理。

### 这一步为什么重要？

因为它把“规范要求”升级成了“硬门槛”：

- 以前主要靠 prompt 约束和 reviewer 提醒
- 现在如果无法证明正式分析基于清洗后的全量数据，该轮根本过不了

这让项目从“希望模型照做”变成了“必须留下可审计证据”。

---

## 8. 第七步：报告、图表与运行轨迹落盘

分析完成后，系统会把关键工件落到 `outputs/run_*/` 目录，包括：

- `final_report.md`
- `cleaned_data.csv`
- 图表
- `agent_trace.json`
- 审稿日志

其中 `agent_trace.json` 很重要，因为它不是简单日志，而是一次运行的“复盘底稿”。

现在里面通常会记录：

- 工作流状态
- 事件流
- analyst 每步摘要
- Python 代码原文
- 阶段执行审计结果
- RAG 摘要与证据登记
- 成功经验 / 失败教训的检索与写回状态
- 审稿历史

---

## 9. 第八步：Review

review 不是“最后润色一下语句”，而是项目的治理层。

当前 review 主要分两层：

- 文本 reviewer
- 可选的视觉 reviewer

review 会重点检查：

- 结论是不是说得太过头
- 图表是否可读、是否支撑结论
- 报告里的知识性解释有没有来源
- 引用与证据是否匹配
- 是否违反了历史中已经反复出现的约束

也就是说，reviewer 更像“审稿人”，不是第二个自由分析助手。

---

## 10. 第九步：记忆写回

这里是最容易被混淆的地方之一。

现在不是“所有东西都写进一个 memory”，而是分层写回：

### 10.1 成功经验写回

只有当一次运行满足这些条件时，才会写入成功经验：

- `review_status == accepted`
- 工作流完整
- 关键工件齐全

为什么这么保守？

因为成功经验是以后会默认拿来复用的内容，必须尽量干净。

### 10.2 失败教训写回

如果一次运行是“完整失败”，则会写入失败教训库。这里的完整失败大致指：

- 有 `final_report.md`
- 有 `agent_trace.json`
- 并且明确失败，比如：
  - 审稿拒绝
  - 达到最大返修轮次仍未通过
  - 阶段执行审计失败
  - 工作流未完成

失败教训只会以负向约束形式参与后续分析，不会被当作事实依据或成功模式。

### 10.3 运行档案永远保留

不管成功还是失败，运行档案都会保留在 `outputs/run_*` 里。

所以现在的沉淀逻辑可以简单记成：

- 运行档案：全留
- 成功经验：只留可信的
- 失败教训：单独留，且只当禁忌清单用
- 外部参考资料：走 RAG，不和运行经验混

---

## 11. 第十步：历史问答

历史问答不是重新分析新数据，而是围绕历史工件继续提问。

它主要读取：

- 历史报告
- 运行轨迹
- 审稿记录
- 图表说明
- 成功经验
- 失败教训

支持两类问题：

### 11.1 单次分析追问

例如：

- 上次为什么用了这个检验？
- 图 2 的意思是什么？
- 哪些结论是数据直接支持的？

### 11.2 跨运行对比

例如：

- 两次类似数据分析的结论有什么变化？
- 哪次运行被拒了，原因是什么？
- 哪些问题反复触发相似的 reviewer 约束？

并且，如果历史问答引用了非 `accepted` 的运行，答案里会强标注它的 `review_status`。

---

## 12. 一句话结论

当前主链路最重要的特征，不是“模型说了什么”，而是：

> 输入、检索、分析、全量数据校验、审稿、记忆沉淀和历史追问，已经被串成了一条可追踪、可解释、可复盘的完整流程。
