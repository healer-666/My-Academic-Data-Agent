# Academic-Data-Agent 项目主链路拆解

这份文档只回答一个问题：

> 当前版本的正式主链到底怎么跑？

## 1. 主链一句话概括

```text
上传表格 -> data_context -> memory recall -> RAG retrieval -> analyst loop -> report / trace -> review -> memory writeback -> history QA
```

## 2. 起点：`run_analysis(...)`

真正的主入口是 [`run_analysis(...)`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/agent_runner.py)。

它负责：

- 加载运行时配置
- 建立 `RunContext`
- 验证并接入结构化表格输入
- 构建 `data_context`
- 触发 Project Memory 回忆
- 触发 RAG 检索
- 运行 analyst loop
- 保存 report / trace
- 触发 review
- 将 accepted run 写回 Project Memory

## 3. 第一阶段：输入标准化

当前正式主线只支持：

- `csv`
- `xls`
- `xlsx`

输入如果是结构化表格：

- 直接进入 `data_context`
- 记录一份 `document_ingestion.json`
- 状态标记为 `not_needed`

说明：

- 仓库中仍保留 `document_ingestion.py`
- 但 PDF 不再是当前默认入口的正式主线

## 4. 第二阶段：数据上下文构建

[`data_context.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/data_context.py) 负责把原始表格压缩成模型能高效消费的摘要，包括：

- 字段名
- 数据类型
- shape
- 前几行样本
- 小样本警告

这一层的目标不是把整份数据塞给模型，而是给模型一个足够紧凑但可操作的数据视图。

## 5. 第三阶段：记忆与检索

### `memory_context`

来自 [`memory/service.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/memory/service.py)：

- 项目级偏好
- 审稿约束
- 历史分析经验

### `knowledge_context`

来自 [`knowledge_context.py`](/C:/Users/pc/OneDrive/Desktop/agent/src/data_analysis_agent/knowledge_context.py)：

- User Intent
- Project Memory
- Reference Context
- Retrieved Knowledge
- Retrieved Evidence Register

## 6. 第四阶段：RAG

当前 RAG 已经是正式主链的一部分。

检索链路大致是：

```text
query builder -> dense retrieval -> keyword retrieval -> merge -> rerank -> final chunks -> evidence register
```

当前重点是：

- 服务表格分析过程中的背景知识补充
- 服务报告中的知识性解释与证据引用
- 作为历史问答的检索底座之一

## 7. 第五阶段：Analyst loop

analyst loop 的职责是：

- 基于结构化上下文做分析决策
- 调用工具
- 回收 observation
- 生成最终报告

当前它不是自由聊天，而是受控分析流程。

## 8. 第六阶段：Report / trace

分析完成后会落地：

- `final_report.md`
- `agent_trace.json`
- figures
- review logs

trace 里会保留：

- workflow state
- event stream
- RAG payload
- evidence coverage
- memory retrieval / writeback
- review history

## 9. 第七阶段：Review

review 分两层：

- 文本 reviewer
- 可选 visual reviewer

review 不只检查语言质量，也检查：

- 结论是否稳妥
- 图表是否可读
- 引用是否缺失
- 证据是否匹配

## 10. 第八阶段：Memory writeback

只有 accepted run 会进入长期 memory：

- 抽取项目级经验
- 精炼成 record
- 按 scope 写入 Project Memory

这样可以尽量减少长期记忆中的噪声。

## 11. 第九阶段：History QA

历史问答不重新执行新的数据分析代码。

它基于历史工件回答问题，包括：

- final report
- agent trace
- review logs
- figure summaries
- Project Memory

当前支持：

- 围绕单次运行继续追问
- 跨多次运行做对比总结

## 12. 一句话结论

当前主链最重要的特征不是“模型回答了什么”，而是：

> 输入、检索、分析、证据、审稿、记忆和历史追问已经被串成了一条可追踪、可治理、可回看的完整流程。
